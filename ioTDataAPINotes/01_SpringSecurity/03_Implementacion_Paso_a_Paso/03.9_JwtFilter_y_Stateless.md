# 1. ¿Por qué necesitamos un filtro?

El login emite el token.

Pero Spring Security no lo procesa automáticamente.

Necesitamos:

- Leer header Authorization
- Extraer token
- Validar firma y expiración
- Cargar usuario
- Colocar Authentication en SecurityContextHolder

Esto se hace mediante un filtro personalizado.

---

# 2. JwtAuthenticationFilter

Debe:

1. Extender OncePerRequestFilter.
2. Ejecutarse una vez por request.
3. Leer header Authorization.
4. Verificar que comience con "Bearer ".
5. Extraer token.
6. Validar token usando JwtService.
7. Crear Authentication autenticada.
8. Colocarla en SecurityContextHolder.

Si el token no existe o es inválido:

- No autentica.
- Deja que Spring responda 401/403.

---

# 3. Registro del Filtro en SecurityFilterChain

Se agrega:

.addFilterBefore(
    jwtAuthenticationFilter,
    UsernamePasswordAuthenticationFilter.class
)

Esto asegura que el token se procese
antes del flujo tradicional de autenticación.

---

# 4. Stateless Definitivo

Debe configurarse:

SessionCreationPolicy.STATELESS

Esto asegura:

- No se crea JSESSIONID.
- No se guarda estado en servidor.
- Cada request es independiente.

---

# 5. Flujo Completo Final

Request →
SecurityFilterChain →
JwtAuthenticationFilter →
JwtService.validate →
SecurityContextHolder.setAuthentication(...) →
Controller

---

# 6. Pruebas Manuales

## Sin token

```bash
curl -i http://localhost:8080/api/ping
````

Resultado esperado:

401 o 403

---

## Con token válido

```bash
curl -i http://localhost:8080/api/ping \
  -H "Authorization: Bearer <JWT>"
```

Resultado esperado:

HTTP/1.1 200

pong

---

## Con token inválido

```bash
curl -i http://localhost:8080/api/ping \
  -H "Authorization: Bearer token_invalido"
```

Resultado esperado:

401 o 403
