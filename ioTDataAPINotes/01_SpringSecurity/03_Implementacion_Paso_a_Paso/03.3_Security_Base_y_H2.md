# 03.3 — Configuración Base de Spring Security y H2 Console

En este capítulo activamos Spring Security
y configuramos el acceso controlado a la aplicación.

---

# Estado esperado al finalizar este capítulo

- Spring Security activo.
- Todas las rutas protegidas por defecto.
- H2 Console accesible.
- CSRF configurado correctamente.
- Aplicación funcional sin errores de frame o CSRF.

---

# Requisitos Previos

Se asume completado:

03.2 — Modelo y JPA

Las entidades y repositorios deben existir.

---

# 1. Activación de Spring Security

Al agregar la dependencia `spring-boot-starter-security`,
Spring protege automáticamente todas las rutas.

Comportamiento por defecto:

- Todas las rutas requieren autenticación.
- Se genera un usuario temporal en consola.
- Se habilita login form.

Este comportamiento será reemplazado progresivamente.

---

# 2. Creación de SecurityConfig

Se define una clase de configuración
que expone un SecurityFilterChain personalizado.

Objetivo:

- Controlar qué rutas son públicas.
- Desactivar configuración automática por defecto.

---

# 3. Permitir H2 Console

La consola H2 requiere:

- Acceso sin autenticación.
- Permitir frames (SAMEORIGIN).
- Ignorar CSRF.

Configuraciones necesarias:

- requestMatchers("/h2-console/**").permitAll()
- csrf().ignoringRequestMatchers("/h2-console/**")
- headers().frameOptions().sameOrigin()

Sin esto, la consola no carga correctamente.

---

# 4. Política de Sesión

En esta etapa aún no usamos JWT,
pero definimos desde ahora:

SessionCreationPolicy.STATELESS

Esto evita que Spring cree sesión HTTP.

La aplicación será completamente stateless.

---

# 5. Configuración General de Autorización

Se establece:

- /h2-console/** → permitido
- Resto → autenticado

Aún no existe endpoint de login personalizado.
Eso vendrá más adelante.

---

# 6. Verificación

Probar:

curl -i http://localhost:8080/api/ping

Resultado esperado:

- 401 o 403 (según entrypoint)
- La ruta está protegida

Acceder a:

http://localhost:8080/h2-console

Debe cargar correctamente.
