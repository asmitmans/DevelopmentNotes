# 1. Autenticación vs Autorización

Es importante distinguir dos conceptos.

Autenticación

→ verificar identidad del usuario.

Autorización

→ verificar si el usuario tiene permisos
para realizar una acción.

JWT resuelve la autenticación.

Los roles resuelven la autorización.

---

# 2. Roles en Spring Security

Spring Security utiliza el concepto de
GrantedAuthority.

Normalmente se utilizan roles con prefijo:

ROLE_USER  
ROLE_ADMIN

Estos se convierten en SimpleGrantedAuthority.

---

# 3. Autorización por endpoint

Podemos restringir endpoints mediante:

requestMatchers()

Ejemplo conceptual:

/api/admin/** → requiere ROLE_ADMIN

/api/user/** → requiere ROLE_USER

Esto se define dentro de SecurityFilterChain.

---

# 4. Flujo de autorización

Request →
JwtAuthenticationFilter →
SecurityContextHolder contiene Authentication →
Spring Security evalúa roles →
Permitir acceso o devolver 403

---

# 5. Beneficio

Esto permite implementar:

- APIs multiusuario
- paneles administrativos
- control de acceso granular

---

Implementar autorización basada en roles
en una API autenticada con JWT.

```markdown
# 03.12 — Refresh Tokens (opcional)

En este capítulo introducimos el concepto de refresh tokens.

Esto permite renovar tokens de acceso
sin requerir que el usuario vuelva a iniciar sesión.

---

# ✅ Estado esperado al finalizar este capítulo

- Access token de corta duración.
- Refresh token de mayor duración.
- Endpoint para renovar access token.

---

# 📌 Requisitos Previos

Se asume completado:

03.11 — Roles y autorización.

---

# 1. Problema con los access tokens

Los JWT normalmente tienen expiración corta.

Ejemplo:

15 minutos  
30 minutos  
1 hora

Esto mejora la seguridad,
pero obliga al usuario a autenticarse nuevamente
cuando el token expira.

---

# 2. Solución: Refresh Token

Se utilizan dos tokens.

Access Token

- duración corta
- usado en cada request

Refresh Token

- duración larga
- usado solo para obtener nuevo access token

---

# 3. Flujo típico

Login →

Servidor devuelve:

access_token  
refresh_token

Cuando access_token expira:

Cliente llama:

POST /auth/refresh

Servidor valida refresh token
y genera nuevo access token.

---

# 4. Ventajas

- Mejor experiencia de usuario
- Mayor seguridad
- Tokens de acceso cortos

---

# 5. Consideración de seguridad

Refresh tokens normalmente:

- se almacenan en base de datos
- pueden revocarse
- tienen mayor control que access tokens

---

# 🎯 Objetivo de este Capítulo

Presentar el mecanismo de refresh tokens
como mejora de seguridad en APIs JWT.
```
