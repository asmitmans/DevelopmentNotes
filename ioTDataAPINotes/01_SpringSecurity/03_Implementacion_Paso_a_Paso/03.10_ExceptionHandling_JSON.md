# 03.10 — Manejo consistente de errores (401 / 403 en JSON)

En este capítulo configuramos Spring Security
para devolver respuestas de error consistentes en formato JSON.

Esto mejora la integración con clientes REST
y evita respuestas HTML o cuerpos vacíos.

---

# Estado esperado al finalizar este capítulo

- Respuestas 401 en formato JSON.
- Respuestas 403 en formato JSON.
- Sin redirecciones a login.
- API completamente RESTful.

---

# Requisitos Previos

Se asume completado:

03.9 — JwtAuthenticationFilter y Stateless

- JWT validado en cada request.
- SecurityContextHolder poblado correctamente.
- Endpoints protegidos funcionando.

---

# 1. Problema con la configuración por defecto

Spring Security, por defecto:

- Puede devolver respuestas HTML.
- Puede devolver cuerpos vacíos.
- Puede usar comportamientos pensados para aplicaciones web.

En una API REST esto no es deseable.

Las respuestas de error deben ser consistentes.

---

# 2. Tipos de errores de seguridad

Existen dos categorías principales.

## 401 — Unauthorized

El usuario **no está autenticado**.

Ejemplos:

- No envió token.
- Token inválido.
- Token expirado.

---

## 403 — Forbidden

El usuario **está autenticado**
pero **no tiene permisos suficientes**.

Ejemplo:

- Endpoint requiere ROLE_ADMIN
- Usuario solo tiene ROLE_USER

---

# 3. Componentes utilizados

Spring Security permite personalizar estas respuestas mediante:

AuthenticationEntryPoint → controla respuestas 401

AccessDeniedHandler → controla respuestas 403

Ambos componentes se registran dentro de SecurityFilterChain.

---

# 4. Formato de respuesta

Las respuestas deben tener un formato consistente.

Ejemplo conceptual:

{
  "timestamp": "...",
  "status": 401,
  "error": "Unauthorized",
  "message": "...",
  "path": "/api/ping"
}

Esto facilita:

- debugging
- logging
- integración con frontend

---

# 5. Flujo con errores

Request →
SecurityFilterChain →
JwtAuthenticationFilter →

Si no existe autenticación válida:

→ AuthenticationEntryPoint (401)

Si existe autenticación pero falla autorización:

→ AccessDeniedHandler (403)

---

# 6. Beneficio Arquitectónico

Con esta configuración:

- La API es completamente RESTful
- Las respuestas son predecibles
- El cliente puede manejar errores fácilmente

---

# Objetivo de este Capítulo

Garantizar respuestas de error consistentes
y apropiadas para una API REST basada en JWT.
